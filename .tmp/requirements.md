# エンジニア向け掲示板サービス要件定義書

## 概要

本サービスは、エンジニア同士の知見共有を目的としたスレッド型フォーラムです。SlackやDiscordなどのクローズドなチャットツールでは情報が埋もれてしまう問題に対して、検索性・公開性・継続的な価値を重視した設計を行います。

Remix + Supabase による構築を前提とし、将来的なスマートフォンアプリ対応を見据えて API ファーストで設計します。

---

## 1. 全体構造

```

ユーザー
└─ コミュニティ（ユーザー作成制）
├─ 公開設定（オープン / リミテッド / クローズド）
├─ メンバー（ロール：オーナー / マネージャー / メンバー）
├─ フォーラム
│   └─ カテゴリー（階層あり）
│       └─ スレッド（Markdown形式）
│           └─ ポスト（返信）
└─ チャット
├─ チャンネル（複数人）
└─ グループ / 1対1 DM

```

---

## 2. アカウントと認証

- 閲覧：ログイン不要（匿名閲覧可能）
- 書き込み：ログイン必須
- ソーシャル認証：
  - Google OAuth
  - GitHub OAuth
- アカウントは全コミュニティで共通利用可能
- 複数コミュニティへの参加・切り替えは Slack 風 UI を想定

---

## 3. コミュニティ機能

### 3.1 作成・管理

- 任意のユーザーが新規コミュニティを作成可能
- 作成者は「オーナー」ロールを持つ

### 3.2 公開設定

| タイプ      | 説明                            | 表示状態     | 参加方法            |
|-------------|---------------------------------|--------------|---------------------|
| オープン    | 誰でも閲覧・参加可能            | 一般公開     | 自由参加             |
| リミテッド  | 承認されたユーザーのみ参加可能  | 一般公開     | 申請→オーナーorマネージャーが承認 |
| クローズド  | 非公開コミュニティ              | 一般非公開   | 招待制               |

### 3.3 権限ロール

| ロール       | 権限                                               |
|--------------|----------------------------------------------------|
| オーナー     | 全権限。公開設定変更、マネージャー追加、メンバー管理 |
| マネージャー | カテゴリ作成、申請承認、チャット管理                |
| メンバー     | スレッド閲覧・投稿、チャット参加                    |

---

## 4. フォーラム機能

### 4.1 カテゴリー

- カテゴリは階層構造をサポート（親カテゴリ→子カテゴリ）
- マネージャー以上の権限で作成・編集可能

### 4.2 スレッド

- Markdown形式で作成
- 画像投稿は可能（動画は非対応）
- LLMによるスレッド要約をスレッド冒頭に表示
- 作成者による編集・削除可（時間制限あり）

### 4.3 ポスト（返信）

- Markdown形式で記述
- メンション（@ユーザー名）、引用対応
- 通知対象（返信・メンション）には通知が送信される

---

## 5. チャット機能

- フォーラムとは別に、非公開チャット機能を提供
- 利用可能範囲：
  - コミュニティメンバー向けチャンネル
  - オーナー・マネージャー限定チャット
  - グループチャット／1対1 DM
- 既読管理あり
- 通知（WebPush / メール）対応

---

## 6. 通知仕様

### 6.1 通知手段

- WebPush（ブラウザ）
- メール通知（設定でON/OFF可能）

### 6.2 通知対象イベント

- メンションされたとき
- 投稿に返信があったとき
- コミュニティ参加申請の承認／拒否
- チャットメッセージ受信時
- モデレーションによる投稿状態変更通知

---

## 7. モデレーション（AIによる）

### 7.1 モデレーション処理

- 投稿完了後に非同期で LLM による判定を実施
- 問題のある投稿は「本人にのみ表示」状態となり、他者からは非表示
- ステータス変更は管理者が手動で実施可能

### 7.2 通報機能

- 全ポスト／スレッド／チャットに通報ボタンを設置
- 通報後、AIモデレーションキューへ送信され処理対象となる

---

## 8. AI要約機能

- 各スレッドごとに、投稿内容を元に LLM により要約生成
- 要約はスレッド冒頭に常時表示
- 要約内容：
  - スレッドの主題・背景
  - 現在の議論の流れ

---

## 9. 翻訳機能

### 9.1 概要

- 投稿・スレッド・コメントなど、全てのテキストに対して**多言語対応**を実施
- 初期対応言語：**日本語 / 英語**
- LLM を活用し、AIによる自動翻訳を行う

### 9.2 ユーザー体験

- 投稿・閲覧・検索すべてを「ユーザーの選択言語」に合わせて操作可能
- 各ポストに「翻訳元言語」「翻訳元表示切り替え」オプションあり
- 将来的には多言語（中国語・韓国語・スペイン語など）へ拡張可能

---

## 10. 検索機能

### 10.1 検索対象と範囲

- キーワードに基づく全文検索を提供
- 検索対象：
  - スレッドタイトル
  - スレッド本文
  - ポスト（返信）
  - カテゴリ名

### 10.2 検索スコープ

| 検索範囲               | 説明                                           |
|------------------------|------------------------------------------------|
| 全体検索               | 参加・未参加を問わず、公開されているすべてのスレッドを対象 |
| 自分の参加コミュニティのみ | 参加しているコミュニティ内全体を対象                     |
| 現在のコミュニティのみ     | 現在表示しているコミュニティ内のみ                       |
| 現在のスレッドのみ         | スレッド内のポスト全文検索                             |

- 各範囲はタブまたはフィルターによって切り替え可能

---

## 11. 技術スタック・設計方針

### 11.1 フロントエンド

- フレームワーク：**Remix**
- SSR 対応：Google 検索インデックスを重視
- モバイルファーストデザイン
- Markdown対応エディタ搭載
- PWA化視野あり

### 11.2 バックエンド / インフラ

- **Supabase**
  - PostgreSQL（主データストア）
  - Storage（画像アップロード）
  - Auth（Google / GitHub OAuth）
  - Realtime（チャットや通知）
- AI処理：
  - LLM（OpenAI API または互換モデル）を活用
  - 投稿モデレーション
  - スレッド要約
  - 多言語翻訳

### 11.3 API設計

- API ファースト
- JSONベースのREST APIまたはGraphQL
- フロントはSSR / クライアント API 両対応
- ネイティブアプリからも利用可能な共通APIを前提に設計
